Java 面试 Redis 查漏补缺

课程未覆盖的高频面试考点

本页梳理了学完《Redis核心技术与实战》40讲之后，在 Java 工程师面试中仍然可能被问到的 Redis 高频知识点。每个主题都包含面试问法、核心答案要点和参考回答，帮你快速查漏补缺。

一、大 Key 问题

面试频率极高，几乎是中高级 Java 面试必问。课程仅零散提及，缺少系统梳理。

什么是大 Key？有什么危害？ 必问

面试场景：聊到 Redis 性能优化、线上故障排查时必然引出

定义标准（业界通用）

  • String 类型：value 大于 10KB 就要注意，超过 1MB 就是典型大 Key
  • 集合类型（Hash/Set/ZSet/List）：元素数量超过 5000 个

四大危害

  • 内存不均：集群中某个节点内存远超其他节点，数据倾斜
  • 阻塞主线程：大 Key 的读写、删除都会耗时长，阻塞其他命令
  • 网络带宽：大 value 传输占满带宽，影响同节点其他 Key 的响应
  • 持久化压力：AOF 写入耗时长，RDB fork 时 copy-on-write 内存暴增

如何发现

# 方法1：官方工具扫描（推荐，不阻塞） redis-cli --bigkeys -i 0.1 # 方法2：查看单个 Key 的内存占用（Redis 4.0+） MEMORY USAGE mykey # 方法3：线上监控，通过 slowlog 发现异常命令 SLOWLOG GET 10

如何解决

  • 拆分：大 Hash 按字段前缀拆成多个小 Hash；大 List 按时间范围分段
  • 异步删除：使用 UNLINK 代替 DEL（Redis 4.0+），后台线程回收内存
  • 开启 lazy-free：配置 lazyfree-lazy-expire yes、lazyfree-lazy-server-del yes
  • 压缩：String 类型可在客户端用 GZIP/Snappy 压缩后再存入
你们线上有遇到过大 Key 问题吗？怎么处理的？
我们遇到过一个场景：用户画像用一个大 Hash 存储，字段超过了 2 万个，导致集群某个分片内存倾斜 40%。发现方式是通过 redis-cli --bigkeys 扫描。解决方案是按用户 ID 取模把一个 Hash 拆成 16 个小 Hash，key 格式改为 user:{uid}:{uid%16}，同时启用了 lazyfree 来避免删除旧 Key 时阻塞。

二、热 Key 问题

与大 Key 并列的高频考点。课程完全没有系统讲解。

什么是热 Key？怎么解决？ 必问

面试场景：秒杀、热搜、明星出轨等突发流量场景

定义

某个 Key 的访问频率远超其他 Key（QPS 可能达到数万甚至数十万），导致该 Key 所在的 Redis 节点成为瓶颈。

发现方式

  • redis-cli --hotkeys（需开启 LFU 淘汰策略）
  • MONITOR 命令（仅在测试环境用，线上会严重降低性能）
  • 客户端/代理层统计：在 Redisson/Lettuce 中做 Key 访问计数
  • 业务层预判：秒杀商品 ID、热搜话题 ID 提前可知

四种解决方案

方案原理适用场景
本地缓存打底用 Caffeine/Guava Cache 缓存热 Key，不请求 Redis最常用，效果最好
Key 分散（读副本）将 key 复制为 key_1~key_N，随机读不同副本读多写少场景
读写分离从节点分担读流量主从架构已部署时
Proxy 层限流在 Codis/Twemproxy 层对单 Key 限流有代理层架构时
面试标准回答模板：本地缓存（Caffeine）+ Redis 两级缓存。热 Key 在 JVM 内存中缓存 1-3 秒，过期后再查 Redis。这样单机就能扛住数万 QPS，极大减轻 Redis 压力。

三、延迟双删策略

缓存一致性的经典追问。课程讲了 Cache-Aside 但没讲延迟双删。

先删缓存再更新数据库，还是反过来？ 必问

面试场景：聊到缓存与数据库一致性时，必然深入追问

三种策略对比

策略步骤问题
先删缓存，再更新DB1.删缓存 → 2.更新DB并发下：线程B在步骤1和2之间读到旧DB值写入缓存，导致不一致
先更新DB，再删缓存1.更新DB → 2.删缓存推荐方案。极端并发下仍有短暂不一致窗口，但概率极低
延迟双删1.删缓存 → 2.更新DB → 3.延迟N毫秒 → 4.再删缓存兜底方案，覆盖并发不一致窗口

延迟双删流程图

// Java 伪代码 public void update(String key, Object value) { // 第一次删除缓存 redis.del(key); // 更新数据库 db.update(value); // 延迟后第二次删除（兜底） executor.schedule(() -> redis.del(key), 500, TimeUnit.MILLISECONDS); }

延迟时间怎么定？

  • 延迟时间 = 读业务耗时 + 几百毫秒（确保并发读请求已完成）
  • 通常设 300ms ~ 1s，根据业务读 DB 的平均耗时评估
面试结论：推荐「先更新DB，再删缓存」（Cache-Aside）。如果对一致性要求高，可以加延迟双删做兜底。如果要求强一致，考虑 Canal 监听 binlog 异步删缓存。

四、Redlock 争议

用 Redisson 必问。课程讲了分布式锁原理，但没涉及 Redlock 和社区争议。

Redisson 分布式锁底层是什么？Redlock 可靠吗？ 必问

面试场景：你说熟悉 Redisson，面试官一定追问底层实现和 Redlock 争议

Redisson 单节点锁的实现

  • 底层使用 Lua 脚本保证加锁原子性（SETNX + EXPIRE 合为一个 Lua）
  • 支持可重入：用 Hash 结构存储，field 为线程ID，value 为重入次数
  • 看门狗机制（Watchdog）：加锁成功后启动后台线程，每隔 lockWatchdogTimeout/3 （默认10秒）自动续期，避免业务未执行完锁就过期
-- Redisson 加锁 Lua 脚本核心逻辑 if redis.call('exists', KEYS[1]) == 0 then redis.call('hincrby', KEYS[1], ARGV[2], 1) -- ARGV[2] = 线程ID redis.call('pexpire', KEYS[1], ARGV[1]) -- ARGV[1] = 过期时间 return nil end if redis.call('hexists', KEYS[1], ARGV[2]) == 1 then redis.call('hincrby', KEYS[1], ARGV[2], 1) -- 可重入：计数+1 redis.call('pexpire', KEYS[1], ARGV[1]) return nil end return redis.call('pttl', KEYS[1]) -- 加锁失败，返回剩余时间

Redlock 算法

Redis 作者 Antirez 提出的多节点分布式锁方案：

  1. 向 N 个独立 Redis 实例（推荐5个）依次加锁
  2. 如果在超时时间内，成功获取 超过半数（≥N/2+1）实例的锁，则加锁成功
  3. 锁的有效时间 = 初始有效时间 - 加锁消耗时间
  4. 解锁时向所有实例发送 DEL

Martin Kleppmann 的质疑

  • 时钟跳跃问题：如果某个 Redis 节点发生时钟跳跃（NTP 校时），锁可能提前过期，导致两个客户端同时持有锁
  • GC 暂停问题：客户端获取锁后遭遇长时间 GC 停顿（STW），锁过期了但客户端不知道，继续操作共享资源
  • 建议：如果需要强一致性，应该使用带 fencing token 的方案（ZooKeeper/etcd）

Antirez 的回应

  • 时钟跳跃可以通过运维手段避免（禁止大幅度 NTP 调整）
  • GC 暂停问题不是 Redlock 特有的，任何分布式锁都有这个问题
那实际项目中你怎么选择？
对于大多数业务场景，Redisson 单节点锁 + 看门狗续期已经足够（Redis 本身是主从+哨兵高可用）。只有在极端要求"绝对不能有两个客户端同时拿到锁"的金融/支付场景，才需要考虑 ZooKeeper/etcd。Redlock 在实际生产中用得不多，因为部署5个独立实例的成本太高，且争议未解。

五、Redis 与 Memcached 对比

经典面试题，课程完全没涉及。

Redis 和 Memcached 有什么区别？ 高频

维度RedisMemcached
数据结构String、Hash、List、Set、ZSet、Stream 等丰富类型只支持简单的 Key-Value（String）
持久化支持 AOF + RDB不支持，纯内存，重启数据丢失
线程模型单线程命令执行（6.0 多线程网络IO）多线程
集群方案原生 Cluster、哨兵客户端一致性哈希
内存管理使用 jemalloc，有内存碎片问题但可优化Slab Allocation，内存碎片少但有 Slab 浪费
事务支持 MULTI/EXEC、Lua 脚本支持 CAS（compare-and-swap）
发布订阅支持 Pub/Sub、Stream不支持
适用场景功能丰富，适合大多数场景纯缓存场景，多线程利用多核更好
一句话总结：Memcached 是纯缓存，简单但高效；Redis 是"瑞士军刀"，功能强大但架构更复杂。现在 Java 项目基本都选 Redis，除非是极简缓存场景且需要多线程打满 CPU。

六、Lua 脚本 vs 事务

课程讲了事务的 ACID 分析，但没有和 Lua 脚本做对比。面试常问。

为什么推荐 Lua 脚本而不是 MULTI/EXEC？ 高频

维度MULTI/EXEC 事务Lua 脚本
原子性命令排队，EXEC 时批量执行，但不支持回滚整个脚本作为一个命令原子执行
中间结果命令之间无法使用前一个命令的返回值可以使用中间变量做条件判断
网络开销每个命令一次网络往返一次网络往返传输整个脚本
错误处理某条命令失败，其他命令照常执行（不回滚）脚本报错则整体不执行
CAS 乐观锁需要配合 WATCH 实现不需要，天然原子

典型使用场景：库存扣减

-- 原子性库存扣减 Lua 脚本 local stock = tonumber(redis.call('GET', KEYS[1])) if stock and stock > 0 then redis.call('DECR', KEYS[1]) return 1 -- 扣减成功 end return 0 -- 库存不足
// Java 调用方式 String script = "local stock = tonumber(redis.call('GET', KEYS[1])) ..."; Long result = redisTemplate.execute( new DefaultRedisScript<>(script, Long.class), Collections.singletonList("stock:1001") );
面试结论：实际开发中几乎不用 MULTI/EXEC 事务，用 Lua 脚本代替。因为 Lua 脚本能做条件判断、性能更好、真正原子。Redisson 的分布式锁底层也是 Lua 脚本实现的。

七、Pipeline 批量操作

实际开发高频使用，课程没有讲。

Pipeline 是什么？和 MGET 有什么区别？ 高频

Pipeline 原理

客户端将多个命令打包一次性发送给 Redis，Redis 顺序执行后将结果一次性返回。核心收益是减少网络往返次数（RTT）。

  • 不使用 Pipeline：100 个命令 = 100 次 RTT（假设 RTT=1ms，总耗时 ≈ 100ms）
  • 使用 Pipeline：100 个命令 = 1 次 RTT（总耗时 ≈ 1ms + 命令执行时间）

Pipeline vs MGET/MSET

维度PipelineMGET/MSET
原子性非原子，命令可能被其他客户端插入原子操作
命令类型可以混合不同类型的命令只能批量 GET/SET
Cluster需要按 slot 分组发送同样需要 key 在同一 slot

Java 示例

// Spring Data Redis Pipeline 示例 List<Object> results = redisTemplate.executePipelined( (RedisCallback<Object>) connection -> { for (int i = 0; i < 1000; i++) { connection.stringCommands().get( ("user:" + i).getBytes() ); } return null; } );
注意事项：Pipeline 打包的命令数不宜过多（建议每批 ≤ 500），太多会占用大量内存缓冲区，也会阻塞后续命令。

八、Pub/Sub 发布订阅

Redis 的消息通知机制，课程完全没讲。面试偶尔会问到。

Redis Pub/Sub 怎么用？有什么局限？ 了解

基本模型

  • 生产者通过 PUBLISH channel message 发送消息
  • 消费者通过 SUBSCRIBE channel 订阅频道
  • 支持 PSUBSCRIBE pattern* 模式匹配订阅

致命缺陷

  • 消息不持久化：发布时如果没有订阅者在线，消息直接丢失
  • 没有 ACK 机制：无法确认消费者是否成功处理
  • 没有消费者组：无法多消费者分担负载（7.0 的 Sharded Pub/Sub 改善了集群场景）
  • 客户端断线重连后丢消息

适用场景

  • 实时通知（不要求可靠性）：聊天室、配置变更通知、缓存失效广播
  • 如果要可靠消息，应该用 Redis Stream 或 RocketMQ/Kafka
Redis 适合做消息队列吗？
看场景。如果是轻量级、允许少量丢失的场景（如通知、排行榜更新），可以用 Redis Stream（5.0+），它支持消费者组和 ACK。但如果是订单、支付等对可靠性要求极高的场景，应该用 RocketMQ 或 Kafka，因为它们有完善的持久化、重试、死信队列机制。Pub/Sub 不适合做消息队列，只适合做广播通知。

九、Redis 7.0+ 新特性

课程截止到 6.0。面试问"Redis最新版本有什么特性"时的加分项。

Redis 7.0 有哪些重要更新？ 加分项

核心新特性

  • Redis Functions：取代 EVAL 的 Lua 脚本管理方式。函数随数据持久化、主从复制，不再需要客户端每次传输脚本。通过 FUNCTION LOAD 加载，FCALL 调用
  • Multi-part AOF：AOF 不再是单个大文件，拆分为 base 文件 + 增量文件 + manifest 清单。AOF 重写不再需要 fork 完整的 RDB，效率大幅提升
  • Sharded Pub/Sub：消息只在 key 所属的 slot 节点内传播，而非广播到所有节点。解决了 Cluster 模式下 Pub/Sub 的带宽浪费问题
  • ACL v2：支持按 Pub/Sub 频道的权限控制，更细粒度的安全管理
  • 命令拆分：拆分出子命令（如 CLIENT NO-EVICT），更清晰的命令语义
  • Listpack 全面替代 Ziplist：解决 Ziplist 级联更新的性能问题
面试怎么说：不需要每个都深入，但能说出 "Multi-part AOF 解决了 AOF 重写的性能问题" 和 "Functions 让 Lua 脚本可以持久化和复制" 就已经足够让面试官眼前一亮了。

十、"Redis 为什么这么快"完整版回答

面试最高频的开场问题。课程讲了一部分，这里给出面试标准的完整话术。

Redis 为什么能达到 10万+ QPS？ 必问

六个核心原因（按面试推荐顺序）

  1. 纯内存操作：数据存在内存中，读写都是内存级别的微秒操作，比磁盘快几个数量级
  2. 高效数据结构：底层使用了 SDS、ziplist/listpack、skiplist、intset 等紧凑编码结构，减少内存分配和 CPU 消耗
  3. 单线程模型：避免了多线程的上下文切换和锁竞争开销。注意：这里的单线程是指命令执行在单线程中，网络IO（6.0+）是多线程的
  4. IO 多路复用：使用 epoll/kqueue 机制，一个线程可以同时监听大量客户端连接，非阻塞 IO
  5. 高效的通信协议：RESP 协议简单，解析快，开销小
  6. 渐进式设计：rehash 渐进式执行、大 Key 异步删除（lazy-free）等，避免长时间阻塞
那 Redis 6.0 为什么引入多线程？引入后还是单线程吗？
Redis 6.0 的多线程仅用于网络 IO 的读写（读取请求、发送响应），命令执行仍然是单线程的。引入原因是网络 IO 成为了性能瓶颈，尤其在高并发、大量小命令的场景下，单线程处理网络 IO 跟不上。多线程网络 IO 能显著提升吞吐量（官方数据：性能提升约一倍），同时保持了命令执行的单线程模型，不需要处理并发安全问题。默认不开启，需配置 io-threads。

小结

以上 10 个专题覆盖了课程之外、Java 面试中最常被问到的 Redis 知识点。建议按优先级复习：

  1. 必问三件套：大 Key + 热 Key + 延迟双删 —— 每次面试几乎都会涉及
  2. Redisson 深度：Redlock 争议 + 看门狗机制 —— 说了熟悉 Redisson 必追问
  3. "为什么快"标准答案：6个点按顺序说清，这是开场第一问
  4. Lua 脚本 > MULTI/EXEC：能现场写一个库存扣减的 Lua 脚本就是加分项
  5. Redis 7.0：知道 Multi-part AOF 和 Functions 就够
